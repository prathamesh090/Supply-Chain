# Supplier Risk Assessment (SRA) Database Schema

This document describes the database structures used by the Supplier Risk Assessment system.

The backend uses **two storage systems**:

1. **SQLite** — for operational risk events and inherent risk calculations
2. **MySQL** — for application data (users, companies, authentication)

This separation ensures that high-frequency risk events do not impact transactional user data.

---

# 1. SQLite Database (Operational Risk Storage)

SQLite is used for storing **inherent risk events** generated by the Inherent Risk Engine.

Database file:

```
Backend/plastic_inherent_risk/plastic_risk.db
```

---

# 1.1 risk_event_history

This table stores operational incidents affecting suppliers.

Each event represents a risk signal generated by the system.

### Table Structure

| Column      | Type      | Description                |
| ----------- | --------- | -------------------------- |
| id          | INTEGER   | Primary key                |
| supplier_id | TEXT      | Unique supplier identifier |
| event_text  | TEXT      | Raw event description      |
| category    | TEXT      | Risk category              |
| risk_level  | TEXT      | Risk severity level        |
| risk_score  | REAL      | Raw risk score             |
| source      | TEXT      | Event source               |
| created_at  | TIMESTAMP | Event creation time        |

### Example Record

```
id: 42
supplier_id: PLASTIC-19749e1e
event_text: Explosion at BASF polymer production plant
category: Safety & Chemical
risk_level: High
risk_score: 82
source: inherent_ai
created_at: 2026-03-11T17:37:33
```

---

# 1.2 inherent_risk_results (Optional)

Depending on implementation, normalized and decayed scores may be stored separately.

Typical fields:

| Column           | Description             |
| ---------------- | ----------------------- |
| normalized_score | standardized risk score |
| decayed_score    | time-decayed score      |
| decay_lambda     | decay factor            |

These values help compute rolling risk impact over time.

# 2. Global Risk Events

Global geopolitical events are stored in **memory** rather than a database.

Reason:

* These events change infrequently
* They are loaded during application startup
* Persistence is not required for simulation scenarios

Events are loaded from:

```
risk_global/event_loader.py
```

Example structure:

```
event_id
event_type
risk_score
risk_level
affected_regions
affects
valid_from
valid_to
```

Example event:

```
event_id: GEO-001
event_type: Trade Sanctions
affected_regions: Europe
risk_level: High
```

# 3. Supplier Registry

Supplier metadata is managed by the Supplier Registry module.

This registry maps suppliers to identifiers used across the system.

Typical supplier attributes:

| Field         | Description                |
| ------------- | -------------------------- |
| supplier_id   | Unique supplier identifier |
| supplier_name | Supplier company name      |
| region        | Geographic region          |
| material      | Product material type      |
| tier          | Supply chain tier level    |

Example:

```
supplier_id: PLASTIC-19749e1e
supplier_name: BASF
region: Europe
material: Polypropylene
tier: Tier-1
```

The `supplier_id` is used as the **primary reference key** across risk engines.

# 4. MySQL Database (Application Data)

MySQL is used for user accounts and company verification.

Connection configuration is defined in:

```
main.py
```

Environment variables:

```
DB_HOST
DB_USER
DB_PASSWORD
DB_NAME
DB_PORT
```

---

# 4.1 users

Stores registered platform users.

| Column        | Type      | Description           |
| ------------- | --------- | --------------------- |
| id            | INT       | Primary key           |
| email         | VARCHAR   | User email            |
| password_hash | VARCHAR   | SHA-256 password hash |
| full_name     | VARCHAR   | User full name        |
| role          | ENUM      | user or admin         |
| is_active     | BOOLEAN   | Account status        |
| created_at    | TIMESTAMP | Account creation time |

---

# 4.2 companies

Stores company verification data.

| Column              | Type    | Description                   |
| ------------------- | ------- | ----------------------------- |
| id                  | INT     | Primary key                   |
| company_name        | VARCHAR | Company name                  |
| industry            | VARCHAR | Industry type                 |
| country_code        | VARCHAR | Country identifier            |
| gstin               | VARCHAR | Indian tax identifier         |
| pan                 | VARCHAR | PAN number                    |
| verification_status | ENUM    | pending / verified / rejected |
| verification_data   | JSON    | verification metadata         |

---

# 4.3 user_company

Maps users to companies.

| Column     | Type    | Description          |
| ---------- | ------- | -------------------- |
| user_id    | INT     | User reference       |
| company_id | INT     | Company reference    |
| is_primary | BOOLEAN | Primary company flag |

Primary key:

```
(user_id, company_id)
```

# 5. Data Ownership by Module

| Module                 | Storage             |
| ---------------------- | ------------------- |
| Supplier Registry      | In-memory registry  |
| Inherent Risk Engine   | SQLite              |
| Financial Risk Engine  | ML model inference  |

---

# 7. Database Design Considerations

### Lightweight Event Storage

SQLite allows high-frequency event logging without external database dependencies.

### Separation of Concerns

Operational risk data is separated from application authentication data.

### Extensibility

Additional tables can be introduced for:

```
supplier history
risk trends
event analytics
```

without affecting existing modules.
